# Music Creator Studio (MCS) — Build Roadmap v1.0

---

## What This Document Is

This is the official build roadmap for the Music Creator Studio (MCS), a closed-loop adaptive music generation control system. It is written to be handed to any Claude instance as a complete reference. All architecture decisions are locked. All design debates are resolved. This document skips the reasoning history and goes straight to what to build, in what order, and why.

---

## System Identity

**MCS is not:**
- An AI music generator
- A content automation pipeline
- A mass production system

**MCS is:**
A personal music studio where Python and AI generate structured prompts, the human executes those prompts manually via Gemini, and the system handles all analysis, assembly, diversity filtering, and state management. Only one song is published at a time. Quality over volume. Human perception is the final authority at every decision boundary.

---

## Locked Architecture Reference

**Full architecture:** MCS Architecture v2.1 (with v2.1 amendments)

**Core philosophy:** Constrain → Generate → Evaluate → Re-constrain (closed feedback loop, not linear pipeline)

**Tech stack:**
- Backend: Python 3.11
- Scheduler: GitHub Actions
- Frontend: HTML + JavaScript, GitHub Pages
- Audio analysis: librosa
- Audio assembly: pydub
- Storage: Google Drive API (3 accounts = 45GB), GitHub repo JSON (state authority), GitHub Releases (final archive)
- Notifications: Email (same pattern as MRE)
- Separate GitHub account from MRE (independent 2,000 Actions minutes/month)

**Hard rules (never violate):**
1. GitHub JSON = single source of truth. Drive and Releases are asset stores only.
2. Perceptual Score P = (vibe, originality, coherence) is final authority over all filters.
3. Entropy injection is mandatory every generation cycle — at least one dimension randomly seeded.
4. Every module writes checkpoint to session JSON before passing control.
5. Stems are never merged until Module 14 Final Export is manually triggered.
6. No new module added without replacing or merging an existing one.
7. Every audio asset uploaded to Drive or Releases gets SHA256 hash stored in session JSON.
8. Only final approved artifacts are committed. Stems and intermediate renders stay local until approved.

## Prompt System Specification (Mandatory — All Agents Must Enforce)

These rules govern every prompt generated by Module 3, Module 3.5, and Module 11. No prompt reaches the user queue without satisfying all of them.

---

### Rule P1 — Mandatory Prompt Schema

Every stem prompt must contain exactly these 8 fields in this order:

```
[1] ROLE
[2] STEM FUNCTION
[3] SONG IDENTITY LOCK
[4] STRUCTURAL CONSTRAINTS
[5] AUDIO CHARACTERISTICS
[6] NEGATIVE CONSTRAINTS
[7] ENTROPY INJECTION
[8] OUTPUT SPECIFICATION
```

No field may be omitted. No field may be merged with another. This schema is enforced by Module 3.5 before any prompt reaches the queue.

---

### Rule P2 — Global Context Block (Cross-Stem Coordination)

Every prompt after the first stem of a session must include a GLOBAL CONTEXT block injected before ROLE:

```
GLOBAL CONTEXT:
Active stems already generated:
- [stem_name]: occupies [frequency_range], role [role_description]
- [stem_name]: occupies [frequency_range], role [role_description]

Your stem must NOT conflict with any active stem in frequency, rhythm, or energy role.
```

First stem of a session is exempt (no active stems yet). All subsequent stems are required. This prevents frequency masking and rhythmic conflicts at the generation stage rather than catching them late in Module 7.

---

### Rule P3 — Entropy Formal Definition

Entropy is defined as a structured object, not a free instruction:

```
E = (dimension, intensity, seed)

dimension: one of {identity, rhythm, energy_trajectory, timbre, harmonic}
intensity: float 0.0–1.0
  0.1 = subtle variation
  0.5 = noticeable shift
  0.9 = experimental
seed: integer (logged in session JSON for reproducibility)
```

Exactly one entropy injection per generation cycle. Intensity and seed must be logged in session JSON. The ENTROPY INJECTION field in the prompt must reflect the intensity level explicitly so Gemini understands the degree of variation requested.

---

### Rule P4 — Controlled Vocabulary for Descriptors

Descriptors are split into two layers:

**Layer 1 — Controlled vocabulary (system-defined, stable):**
- mood: {dark, bright, melancholic, tense, euphoric, neutral}
- texture: {grainy, smooth, distorted, clean, warm, cold}
- energy: {sparse, layered, dense, driving, ambient}
- dynamic: {static, building, declining, volatile}

**Layer 2 — Free-form (entropy layer only):**
Used exclusively in the ENTROPY INJECTION field. Unrestricted language allowed here only. All other fields must use controlled vocabulary.

This separation reduces Gemini interpretation variance while preserving creative flexibility where it is intentional.

---

### Rule P5 — Output Normalization (Required in Every Prompt)

The OUTPUT SPECIFICATION field must always include:

```
OUTPUT SPECIFICATION:
- Peak level: -1 dB
- No leading or trailing silence
- Exact loop alignment to bar grid
- No fade in or fade out unless explicitly requested
- Duration: [exact seconds]
```

Module 5 Stem Intake verifies compliance with these specifications on every ingested file.

---

### Rule P6 — Execution Priority and Dependency Ordering

Every prompt in the queue must carry:

```
execution_priority: 1 | 2 | 3
  1 = foundational (drums, bass) — generate first
  2 = harmonic (chords, melody, pads) — generate second
  3 = texture/refinement (effects, layers, vocals) — generate last

dependency_flag: [list of stem IDs that must exist before this prompt is executed]
```

The frontend prompt queue displays prompts in priority order. You always generate foundational stems first so that subsequent stems can reference them in the GLOBAL CONTEXT block accurately.

---

### Rule P7 — Regeneration Protocol

Any stem that fails Module 5, 7, or 8 checks must trigger a structured regeneration prompt, never a re-run of the original prompt. The regeneration prompt must include:

```
REGENERATION CONTEXT:
Previous attempt failed due to: [specific reason from module flag]
Adjustment required: [specific correction]
All original constraints remain unless explicitly modified below.
```

Random retries are prohibited. Every retry must be diagnostically informed.

---

### Rule P8 — Prompt Metadata Logging

Every generated prompt must log the following to session JSON before entering the queue:

```
prompt_metadata:
  - structure_version: [schema version number]
  - entropy_config: {dimension, intensity, seed}
  - descriptor_set: [list of controlled vocabulary terms used]
  - constraint_density_score: [count of active constraints]
  - global_context_stem_count: [number of stems in context block]
```

This metadata is linked to the session's Perceptual Score P at review time. Over time this enables prompt optimization — identifying which structures and entropy configurations produce high vibe, originality, and coherence scores.

---

### Module 3.5 — Prompt Validator (NEW)

Runs after Module 3 and Module 11 generate prompts, before Module 4 sends the notification.

**Validation checks (in order):**
1. All 8 schema fields present and non-empty
2. No field contradictions (example: stem function = percussion + negative constraint = no rhythm)
3. BPM defined and within Style Anchor range
4. Loop length defined
5. Entropy applied exactly once per cycle
6. Global context block present for all non-first stems
7. Output normalization fields complete
8. Execution priority and dependency flags set

**On validation failure:**
- Contradictions → auto-flagged with specific conflict described, presented to user for manual resolution
- Missing fields → auto-filled with session defaults where safe, flagged where not
- Invalid entropy → regenerated with correct bounds

No prompt reaches the queue with an unresolved validation failure.

## Foundational Vectors (Reference)

**Song Identity Vector (per session):**
`I = (mood, energy_trajectory, density_trajectory, harmonic_center, energy_slope, peak_timing, density_volatility)`

**Style Anchor Vector (system-wide):**
`A = (genre_range, tempo_range, energy_profile, rhythmic_signature)`
Never updated automatically. Only updated by explicit user recalibration.

**Diversity Feature Vector (per song):**
`D = (MFCC, tempo, spectral_centroid, chroma, energy_curve_shape, density_curve_shape, harmonic_movement_shape, beat_interval_distribution)`

**Perceptual Score (human input):**
`P = (vibe, originality, coherence)` + binary flag: safe | stretch

**Entropy Quota:**
At least one dimension from {identity, rhythm, energy_trajectory} randomly seeded per cycle. Logged in session JSON.

## System States

**STATE 1 — Candidate Generation (default)**
Runs every weekday. Generates one song plan per day. Ends when at least one candidate is approved at end-of-week review. If no candidate approved, State 1 repeats next week.

**STATE 2 — Refinement**
Activates when at least one candidate approved. Daily generation stops. Multiple active projects allowed simultaneously, worked in any order. Returns to State 1 only on manual trigger after all projects marked finished.

## Unverified Assumptions (Test Before Building)

These must be verified in a local test environment before any module that depends on them is built:

1. **Pydub stem sync** — Can separately generated Gemini stems at the same BPM be layered without drift?
2. **Sample Extender quality** — Does librosa segmentation on Gemini output produce clean loop points?
3. **Gemini prompt variation** — Does varying creative descriptor language produce meaningfully different audio?
4. **Google Drive API multi-account** — Can one Python pipeline authenticate and write across 3 Google accounts?
5. **GitHub Actions audio processing** — Does librosa + pydub run within Actions memory and time limits?

---

## Deferred to v3.0 (Do Not Build Yet)

These are known future needs, deferred deliberately because they require a populated song library to be meaningful:

- **Strategic exploration targeting** — Module 1 upgrade to target most underrepresented feature space corner, not just random entropy
- **Exploration Score** — Second perceptual signal (novelty/interest) to complement P and prevent long-term aesthetic narrowing

Flag these for v3.0 after 10+ approved songs exist in the library.

## PHASE 10 — Algorithmic Video Generation + YouTube Publish
**Status: Locked for discussion after music production pipeline confirmed ready (Phase 9 complete)**
**Goal:** Auto-generate a visual music video from the song's own audio data and publish directly to YouTube.

**Concept:**
All librosa analysis data computed during audio processing (energy curve, spectral centroid, beat grid, MFCC, frequency bands) is reused as animation input. No new analysis needed. The video is mathematically derived from the music itself — visuals are a direct representation of the song's sonic identity.

**Planned approach:**
- moviepy as the video assembly library (Python, free, no external dependencies)
- Visual elements driven by existing librosa data: waveform visualizer, frequency spectrum bars, energy pulse effects, beat-synced transitions
- Final video = audio track + algorithmically generated visuals, assembled in Python
- YouTube Data API v3 for one-button publish: title, description, tags, AI disclosure flag, all automated
- Fits as an optional extension of Module 14 Final Export

**Why Path A (algorithmic) over Path B (AI video generation):**
- Zero new external tools or manual handoffs
- Visuals derived from the song's own math — coherent with MCS philosophy
- No additional terms of service or generation costs
- librosa data already computed — the video layer is essentially free to generate

**What this phase covers when scoped:**
- moviepy integration and visual template design
- Energy/spectrum animation system driven by librosa output
- Beat-sync engine for visual transitions
- YouTube API authentication and upload automation
- Publish flow integration into Module 14

**Do not scope or build until:** Phase 9 completion gate passed and at least one real song published manually to verify the full pipeline works end to end.
